Language Features

Basic Syntax

Comments

Eugene supports line and block comments.

Examples:

// This is a line comment


/* This is

a block comment */


Declaration of Primitives/Variables

Eugene supports three types of variables -- boolean, txt, and num. A variable of type boolean can have one of the two values: true or false. A txt variable is a string of characters, and a num variable can store a number. Multiple declarations of the same type can be made on one line by separating assignments by commas.

boolean b;
txt txt1,txt2, txt3;
num num1,num2, num3;


Declaration of Arrays

Eugene supports txt[] arrays, which is a list of txt values, and num[] arrays, which is a list of num values. The txt or num values of an array can be accessed by specifying an index between squared brackets. The first element of an array is indexed with zero (0).

txt[] arrayTxt1, arrayTxt2;
num[] arrayNum;


Expressions and Assignments

Eugene supports the following types of operations: The + operator can be used to add numbers, concatenate txt variables or values, and to concatenate arrays. The –, *, and – operators can be only be used to subtract, multiply, and divide num variables or values respectively.

The = assignment operator sets the value of the destination to the source, as long as types match. In the previous examples, the sources were primitive values, references to primitive values, references to property values of parts, and references to property values of devices. The destinations were all references to primitives; however, references to part property values can be set like this, as well. A property can be set to a property even if the property names are not the same, as long as types match. The properties of devices cannot be set. Parts and devices cannot be set to references to other parts and devices. Assignments of values to variables can be written at the declaration of a variable or an array, as well as everywhere in the entire Eugene script. If the variable or array at the left side of the assignment is not declared, the variable will get assigned the value and type of the right side of the assignment.

// the myTxtArray is not declared and will get assigned the type txt[]
myTxtArray = ["A", "list", "of", "strings"];
num[] arrayNum2 = [1, 2.0, 3.00, 4, 2.5 + 2.5];
num two = numLst2[1]; // two holds 2.0
num four = two + 2; // four holds 4.0
num 2 = (((44.4 + 6.7) * 2) - 1.45) / 5;
arrayTxt3 = myTxtArray + ["with", "more", "strings"];
arrayNum3 = arryNum2 + [6, 7.0, 8];
txt txt3 = "This", txt4 = "is", txt5 = "a", txt6 = "shortcut";
rbs2.Sequence = "ATCG";
rbs1.Sequence = rbs2.Sequence; // rbs1.Sequence holds "ATCG"

IO Statements - print()

The print() function writes information to the console output and can take multiple comma-separated arguments. Its arguments can either be variables, expressions, or synthetic biological components.

print(“Hello World!”);
print(“Hello” “, world,” “!”);
print(“Device d: “, d);

Data Import and Export

SBOL ("Synthetic Biology Open Language")

Synthetic Biology Open Language (SBOL) is an open-source standard for in silico representation of genetic designs. SBOL is designed to:

▪ Allow synthetic biologists and genetic engineers to electronically exchange designs

▪ Send and receive genetic designs to and from biofabrication centers

▪ Facilitate storage of genetic designs in repositories

▪ Embed genetic designs in publications

SBOL is built around the idea of a core that is used to unambiguously specify the design of a DNA molecule. Around the core are extensions that are used to increase the kind and amount of information transmitted by the language. There are six extensions under development. For example, one extension includes data and information on the performance of DNA components. The adoption of SBOL offers many benefits, including: (1) enabling the use of multiple tools without rewriting designs for each tool, (2) enabling designs to be shared and published in a form other researchers can use even in a different software environment, and (3) ensuring the survival of design (and the intellectual effort put into them) beyond the lifetime of the software or the researchers that were used to create them.

Domain-Specific Control

Declarations of Properties

Properties are significant labels for primitives to be associated with Parts and Part Types. Property definitions must specify a primitive type.

Property Sequence(txt);
Property RelativeStrength(num);
Property Letters(txt[]);
Property Numbers(num[]);
Property Bool(boolean);

Declarations of Components

Eugene provides various types of components of different levels of abstraction, i.e., Device, Part Type, and Part. Devices and Parts have one pre-defined attribute, i.e. sequence, and various pre-defined operations, such as toSequence() or size().

Part Type

Part Types represent types of standard biological parts, such as promoters, ribosome binding sites, or terminators. Part Types do not construct any parts, but rather specify the structure and properties of part that can be constructed. All properties referenced by a Part Type definition must be defined before the Part Type's definition.

Part Promoter(Sequence, RelativeStrength, Letters, Numbers, Bool);
Part RBS(Sequence);
Part Gene(Sequence);
Part Terminator(Sequence);

Device

Devices represent a composite of standard biological parts and/or other devices. In a device declaration, the same part and/or device can be used more than once. Eugene allows synthetic biologist to specify abstract, instantiated, and hybrid synthetic devices. Abstract devices are only assembled of part types, such as promoters, ribosome binding sites, or terminators. Hybrid devices are devices that are assembled of other devices, part types, and parts.

Device emptyDevice();
Device abstractDevice (Promoter, RBS, Gene, Terminator);
Device hybridDevice (abstractDevice, pBad, CI, BBa_B0010, pCI, Reporter, Terminator);

include

Eugene allows to include Eugene scripts into a Eugene script, regardless where in the code. Therefore, the include statement can be used, where the path of the included Eugene scripts must be specified if they are located in a different folder.

The include statement can take multiple Eugene scripts that must be specified in a comma-separated list.

include PropertyDefinition.h;
include PartTypeDefinitions.h, MyFunctions.h;
include ./path/to/MyDevices.eug;

Instantiation of Part Types and Abstract Devices

Parts

Part instantiations define instances of predefined Parts Types and assign concrete values to the Part Type's properties. Eugene provides two possibilities to set the property values of Parts:
1.a comma-separated list of values can be specified, by specifying the property values in the same order as specified in the Part Type declaration.
2.the name of the properties can be specified to which the given value should by assigned to.
The property values of a part can be accessed using the 'dot' operator by first specifying the part name, followed by a dot ('.'), and the desired name of the property.

txt seq = "ATCG";
Promoter p1(seq, 100, ["A", "B", "C"], [1, 2, 3], true);
Promoter p2(.RelativeStrength(20), .Sequence("GCTA"));
RBS rbs1("CGAT");
RBS rbs2();
Gene RFP("GTCCTATA");
Terminator t1();
txt p1Seq = p1.Sequence; // p1Seq holds "ATCG"

Devices

In Eugene, only abstract devices can be instantiated. Instantiated devices consists of parts, such as pLac or lacI. Each part must be an instance of the corresponding part type in the abstract device.
Device concreteDevice(p2, rbs1, RFP, t1);
Property values of devices can be accessed with the dot operator; however, the value is the union of the property values of its members returned as a list. If the property is a txt or num, a txt[] or a num[] is returned. If the property is a txt[] or a num[], a txt[] or a num[] is also returned that consists of the lists appended together. Individual members can be accessed by using square brackets and specifying an index. The first member is indexed at zero. Square brackets can be stacked in the case of devices within devices.

txt[] d1Seq = d1.Sequence; //d1Seq holds ["ATCG", "ATCG", "CGAT"]
txt rbs1Seq = d2[1][2].Sequence; // rbs1Seq holds "CGAT"

Arrays of Synthetic Biological Components

Eugene offers facilities to declare arrays of Devices, Device Instances, Part Types, Part Instances and Properties. Every array of synthetic biological components can only contain elements of the corresponding biological component, for example, a Device[] array can only consist devices. Similar to num[] and txt[] arrays, the elements of arrays of synthetic biological components can be accessed by using squared brackets, where the index is zero-based. Arrays may access the built in functions get and size.

Property[] arrayProperties = [Property1, Property2, … , PropertyN];
PartType[] arrayPartTypes = [PartType1, PartType2, … , PartTypeN];
Promoter[] arrayPromoters = [Promoter1, Promoter2, … , PromoterN];
Device[] arrayDevices = [Device1, Device2, Device3, … , DeviceN];

Declaring Functions

Equivalently to the C programming language, Eugene provides facilities to declare function prototypes. A function is a named block of Eugene statements that performs a certain task and can return some value to its invoker.

// in the following we declare a function prototype which
// gets as parameter a num[] array and prints its values
// in reverse order by using Eugene's print() function
function reversePrint(num[] arrayNum) {
if(arrayNum.size()-1>=0) {
for(num i=arrayNum.size()-1;i>=0;i++) {
print(“arrayNum[“,i,”]: ”,arrayName[i]);
}
}
}

Calling a Function

In order to execute a function prototype, the function must be called. In Eugene, a function can be executed (called) multiple times, from various places during a single execution of a Eugene script.
Note: Currently, Eugene does not support recursive function calls.

// call the reversePrint() function to print the values
// of the arrayNum2 array in reverse order
reversePrint(arrayNum2);

Control Flow Statements

Eugene scripts are generally executed in a top-down manner. Control-flow statements break up the flow of execution and allow blocks of code to be executed conditionally or as long as a condition is fulfilled. Conditions are value comparisons of primitives, biological properties and components, as well as evaluations of rules. For value comparisons, Eugene offers the following relational operators:
• == ... equal to
• != ... not equal to
• > … greater than
• >= ... greater or equal to
• <= ... less or equal than
• < … less than

In Eugene, only values of the same type can be compared. If the values are arrays, then only the == and != operators are supported, where each element in one list is compared to the element at the corresponding position in the other list.
Conditions can be:
• combined by using the AND, OR, or XOR logical operators,
• negated by using the NOT operator, and
• grouped by using parenthesis.

IF-ELSE Statements

Eugene offers conditional branches (i.e., IF-ELSE statements) to manage the execution of specific actions under certain conditions. IF-ELSE statements evaluate the specified condition, where the evaluation result is a boolean value – true or false. If the condition results in true, then the branch specified under the IF condition gets executed. If the condition results in false, then the ELSE branch is executed. It is planned to integrate ELSEIF branches in the near future.

num x = 4;
if (p1.RelativeStrength > x AND NOT(p1.Sequence == "ATCG")) {
p1.Sequence = "TCGA";
} else {
p1.Sequence = "CGAT";
}
if (on(d1, d2, d3) (r1 AND r2) OR (NOT(r2 OR r3))) {
RBS rbs3("GATC");
} else {
RBS rbs3("ATCG");
}

Loops

To iterate over a set of synthetic biological components or to repeat an execution multiple times. Equivalent to Java, Eugene offers three variations of loops – FOR, WHILE, and DO-WHILE. Loops may be nested too.

FOR Loops

FOR loops take three arguments which are separated by a semicolon – the initialization, condition, and incrementation arguments.
First, the initialization arguments is executed, followed by an evaluation of the condition argument. If the FOR loop condition evaluates to true, the body of the loop gets executed. After the body's execution, the third argument of the FOR loop, i.e. the incrementation argument, is executed, and the loop condition is evaluated again. The FOR loop terminates if the evaluation of the loop conditions evaluates to false.

// print the numbers from 1 to 10
for(num i=1; i<=10; i++) {
print(“i:”, i);
}
// now, let's print the values of the arrayNum2 array
// in reverse order
for(num X=arrayNum2.size()-1; X>=0; X--) {
print(“arrayNum[“,X,”]:”, arrayNum2[X]);
}
WHILE Loops

WHILE loops evaluate a specified conditional expression, and while the evaluation returns true, the specified statements in the loop body will be executed.

// print the numbers from 1 to 10 using a WHILE loop
num i=1;
while(i<=10) {
print(“i: ”,i);
i++;
}
// now, let's print the values of the arrayNum2 array
// in reverse order by using a WHILE loop
num X=arrayNum2.size()-1;
while(X>=0) {
print(“arrayNum[“,X,”]:”, arrayNum2[X]);
X--;
}
DO-WHILE Loops

Different to the WHILE loop, the DO-WHILE loops evaluates the specified condition after the execution of the loop's body instead of before the execution of the loop body. Therefore, the statements within the DO block are always executed at least once.

// print the numbers from 1 to 10 using a DO-WHILE loop
num i=1;
do {
print(“i: ”, i);
i++;
} while(i<=10);
// now, let's print the values of the arrayNum2 array
// in reverse order by using a DO-WHILE loop
num X=arrayNum2.size()-1;
do {
print(“arrayNum[“,X,”]: ”,arrayNum2[X]);
X--;
} while(X>=0);

Pre-Defined Functions

on Synthetic Biological Components
add(element)
The add() function appends the specified element to the list of a component's elements.
Examples:
// add the Strength Property to the Promoter Part Type
Property Strength(num);
Promoter.add(Strength);
// add the Promoter Part Type to the hybridDevice hybridDevice.add(Promoter);
get(idx)
The get() function has one parameter that specifies the position of the element which should be returned. The indexing of the elements starts with the index 0.
Examples:
// get the first element of the abstractDevice, i.e., Promoter
abstractDevice.get(0);
// get the i-th element (i.e., the 4th) of the arrayDevices
num i = 3;
arrayDevices.get(i);
remove(idx)
The remove() function takes on parameter which specifies to index of the element that should be deleted from the list of elements.
Examples:
// delete the first element of the abstractDevice (i.e. the Promoter)
abstractDevice.remove(0);
// delete the second Property of the Promoter Part Type (i.e., RelativeStrength)
Promoter.remove(1);
size()
The size() function returns the size of the component. The returned value can be assigned or compared to a variable.
Examples:
// the following statement returns 4
abstractDevice.size();
// assign the size of the arrayDevices array to the nSize variable
num nSize = arrayDevices.size();
toSequence()
The toSequence() function returns a txt value that represents the DNA sequence of the biological component. If the toSequence() function is utilized on a device instance, the function returns the DNA sequence of the assembled Device. For abstract and hybrid devices, the toSequence() function inserts placeholders for the Part Types.
Example:
// print the sequence of the concreteDevice to the console
print("DNA sequence of the concreteDevice:", concreteDevice.toSequence());
// assign the sequence of the hybridDevice to the sequence txt variable
txt sequence = hybridDevice.toSequence();

Combinatorial Functions

To automatically generate synthetic biological devices that comply with a given set of rules, two built-in functions have been added to Eugene – permute and product.

Both functions return a list of generated devices. Both functions can take up to three arguments: a device, if the device generation process should include the existing rules or not, and a number of desired variations. Only the first argument – the input device – is required, whereas the second and third arguments are optional. For the second parameter, which specifies the level of rule enforcement, two options can be specified: strict or flexible. The strict option only generates devices that obey the specified rules, while the flexible option, which is default, generates every possible device and just labels devices that violate the rules. The third parameter is a non-negative integer number and limits the number of variations generated. If product or permute is called with a number smaller than the total number of possible variations, it will generate a random subset of the that size. If the number of possible variations is greater than 1000, an error will be thrown, and the user will be called to reduce the number of possible variations through the application of rules to the generated devices.

product()

The product() function generates all possible devices from the information stored in the design space that comply with the specified rules and order. Given a device composed of n components, and a design space with m elements of the device's components, the product() function generate m^n variations of that device. Rules can be used to prune the set of generated devices.

// generate 5 possible rule compliant device instances of the abstractDevice product(abstractDevice, strict, 5);
// generate all devices that adhere to the hybridDevice's structure product(hybridDevice);

permute()

The permute function automates the specification of many devices that share the same structure. It generates a device for every permutation of the parts in the design space. Hence, given a device composed of n parts, the permute function will generate n! permutations of the parts. If a component of the given device is a device, even ones with only one part, it is not changed and appears in every variation.

// generate all permutations of the abstractDevice that satisfy all defined rules;
permute(abstractDevice, strict);

// generate all permutations of the hybridDevice and label the devices which do not satisfy the rules;
permute(hybridDevice, flexible);

Genetic NOR Gate

A genetic NOR gate starts with two inducible promoters, which initiate – dependent on their input signals – an expression of a reporting gene, such as a green fluorescent protein (GFP). Only if both input signals are “off”, the repressor allows a downstream promoter to initiate the reporter. If both input signals are “on” or just one of them, the downstream promoter is repressed and hence, the reporter will not be expressed. A Eugene script which specifies multiple genetic NOR gates can be downloaded here.
